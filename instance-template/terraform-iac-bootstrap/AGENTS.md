# terraform-iac-bootstrap/ — Infrastructure as Code

> **Scope override.** This directory contains Terraform only — no Go tools, no application code.

## An instance commits zero Terraform

Both the roots' `*.tf` and the per-environment `*.tfvars` are **generated** by
`llz render` and gitignored (see `.gitignore` here). The `*.tf` come from the
`tfroots` package embedded in the `llz` binary (`tools/internal/tfroots`); the
`*.tfvars` are rendered from the LandingZone spec, which is the source of truth:

```
landingzone.yaml + environments/<env>.yaml   →  llz render  →  *.tf + <env>.tfvars
```

So a bare clone has no Terraform in it. `llz render` must run before any
`terraform` command — CI does this for you (the `terraform-init` composite action
runs `llz render --tfvars-only`). The only tracked files here are the provider
pins, `.terraform.lock.hcl`, which are per-instance and regenerated by
`terraform init`.

**Consequence:** do not hand-edit a `*.tf` or `<env>.tfvars` — the next render
overwrites it. Change the spec, or change the embedded root upstream in the
template. For the same reason, `terraform fmt` is not a pre-commit concern here:
the files it would format are not committed.

## Layout

Three generated roots:

```
terraform-iac-bootstrap/
  cluster/          LKE-E cluster + node pool + node firewall — one apply per cluster
  object-storage/   Loki / Harbor S3 buckets (Linode Object Storage) — one per deployment
  vpc/              VPC + subnets — applied before the cluster that attaches to it
```

The reusable modules are **not** in this directory. They live at the repo root
(`terraform-modules/llz-cluster`, `llz-pool`, …) and the roots consume them as
pinned `git::` sources at the template version, e.g.

```hcl
source = "git::ssh://git@github.com/<org>/lke-landing-zone.git//terraform-modules/llz-cluster?ref=<llz_version>"
```

`llz upgrade` re-pins that `?ref=` in lockstep with the rest of the scaffold.

## Apply order

`vpc` → `cluster` → `bootstrap-openbao`, with `object-storage` independent of the
cluster chain. In `llz-terraform.yml` these are the `apply-vpc`, `apply-cluster`,
`bootstrap-openbao` and `apply-object-storage` jobs.

There is no `cluster-bootstrap` Terraform root. Installing Akamai App Platform
(apl-core) is `llz ci bootstrap-cluster`, which the `bootstrap-openbao` job runs
at its head so the cluster and OpenBao phases share one container-init +
checkout + control-plane-ACL cycle. OpenBao auth/policy/KV config is likewise not
Terraform — it is `llz ci bao-*`, from the same job.

For two-cluster deployments, complete the **primary** cluster fully before
applying the secondary: the primary bootstrap seeds Harbor robot credentials and
CA certs that the secondary consumes.

## State backend

State is stored in Linode Object Storage via the S3-compatible backend. Every `backend.tf` requires these flags — do not remove them; standard AWS S3 validation fails against Linode:

```hcl
backend "s3" {
  skip_credentials_validation = true
  skip_metadata_api_check     = true
  skip_region_validation      = true
  force_path_style            = true
  ...
}
```

The S3 endpoint cannot be passed via `-backend-config` CLI flags (it is a nested block). Set `AWS_ENDPOINT_URL_S3` as an environment variable instead — the AWS SDK picks it up and Terraform's S3 backend respects it.

Backend credentials are injected via:
- `AWS_ACCESS_KEY_ID` → GitHub secret `TF_STATE_ACCESS_KEY`
- `AWS_SECRET_ACCESS_KEY` → GitHub secret `TF_STATE_SECRET_KEY`
- `AWS_ENDPOINT_URL_S3` → GitHub variable `TF_STATE_ENDPOINT`

Never add these to `backend.tf` directly.

## Common commands

```bash
# Materialise the roots + this env's tfvars from the spec first — nothing below
# works without it.
llz render <env>

# Initialise (supply backend config from secrets — never commit backend.tfvars).
# The key's middle segment is the DEPLOYMENT discriminator (primary | secondary |
# lab | e2e), not a Linode region.
terraform init \
  -backend-config="bucket=platform-terraform-state" \
  -backend-config="key=cluster/<deployment>/terraform.tfstate" \
  -backend-config="region=us-east-1"

# Plan / apply a deployment. <env>.tfvars is generated by `llz render` above.
terraform plan  -var-file=<env>.tfvars
terraform apply -var-file=<env>.tfvars
```

In CI these run as `llz ci tf-plan` / `llz ci tf-apply`, which wrap the same
invocations and handle the render + backend wiring.

## Conventions

- The LKE control-plane ACL is seeded at create from `github_runner_*_cidrs` and then owned by the in-cluster cloud-firewall-controller (Linode API). github.com-hosted runners open their egress IP at runtime via `llz ci runner-acl open`.
- Do not run `terraform state rm`, `terraform state mv`, or `terraform init -reconfigure` without explicit user approval. State corruption is not recoverable without a backup.
- Do not add resources here that apl-core (or Argo CD on top of it) should own. Terraform bootstraps the cluster; everything after that lives in `apl-values/` and is reconciled by apl-core's in-cluster Argo CD.

## What Terraform manages here vs apl-core

| Managed by Terraform | Managed by apl-core (or its Argo CD) |
|----------------------|--------------------------------------|
| LKE-E cluster, node pool, VPC | All Kubernetes workloads |
| Cluster-level Linode Cloud Firewall + node firewall | Helm releases (cert-manager, Harbor, Prometheus, Istio, Keycloak, OpenBao via `manifest/`, etc.) |
| Linode Object Storage buckets (`object-storage/`) | Loki configuration |
| — (apl-core itself is installed by `llz ci bootstrap-cluster`, not Terraform) | Argo CD self-management after bootstrap |
| — | Everything reconciled from `apl-values/<env>/manifest/` |

Do not duplicate resources between the two systems.

## Rules that apply from root

- Never add `Co-Authored-By` to commits.
- Do not make changes without explicit user approval.
- Do not commit real secrets, API tokens, or kubeconfig files.
